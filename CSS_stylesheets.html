Lesson 4

Stylin' and Profilin'

In the world of computer science there is a design principle referred to as 'separation of concerns' (SoC). The underlying thought is that when you design and develop a computer program, you divide it up into separate sections each with its own focus that addresses a specific concern or problem to be solved. You might also have heard it referred to as object oriented design or modularity. In a particularly complex program you might have a 'module' whose only concern is logging the user into their account. You might have a 'module' whose sole concern is printing. You might have a 'module' whose single focus is on searching a database for items. The beauty of this school of thought is that you can reuse these bits and pieces as you see fit to perform this particular task elsewhere in the program and that if you need to update code for a particular task, you only need to update the code in the 'modules' whose focus is that particular task.

In frontend development we use a modified version of that, yeah? We typically separate our concerns into three major components: structure, style, and the interactive. We've covered a bit of this already with html. Do you remember the primary focus of html?

It's structure

Html is the first division and it deals solely in the structuring of content.

CSS is the second division and it deals solely with the styling of content

Javascript is the third division and it deals solely with the interactive functionality of content

This separation of concerns means that for each focus, there is one division that has primary overwatch. There is certainly some overlap, but if you're dealing with content structure, html is where you generally want to be, if you're dealing with content styling, it's CSS, and if it's a matter of dealing with interactive elements, then it's JavaScript. For CSS, particularly as it concerns external style sheets--which we'll cover in a minute--this division of labor has a number of significant advantages:

  • Speed - On a page load, the browser cache's the style sheet. Web pages that use the cached style sheet will load faster because the style information won't need to be downloaded again as it's already in memory
  • Maintenance - This is an extension of that idea of modularity. If I want to change the styling rules for multiple <table> elements across multiple pages, I can do that in one spot--the external style sheet. This saves time, effort, and reduces the chances of errors.
  • Accessiblity - If--for whatever reason--your CSS doesn't work (e.g. maybe you've borked the syntax, maybe the browser being used doesn't recognize it, whatever), CSS is designed to 'gracefully degrade, ' which means the user can still see the content. It won't look like you want it to look, but it will still be there on the page when the page is loaded.
  • Customization - The use of CSS means you can have multiple sites that look completely different, but use the same html markup. The structure underneath doesn't change when using CSS only the way that content is displayed on the screen. This also allows for users to use their own custom style sheets via their web browser (e.g. if I want every instance of a hyperlink to be bright pink and bold on every page I visit, I can do that).
  • Consistency - Because all the styling is handled in one place (i.e. the style sheet) every element that is attached to that style sheet is styled exactly the same. It also means that any additional content I--as the developer--add to the site later on has the exact same rules applied to it and, consequently, is styled exactly the same as the original content.
  • Portability - This one you probably won't have much to do with, but it's worth mentioning. If you want to translate your site to a different medium (e.g. think printer-friendly preview), you can do that simply by applying a different style sheet to the same content. Again, the content itself doesn't need to be changed, only the way it's displayed.

Cascading Style Sheets

Style sheets work by applying a specified rule-set to a particular element--or sets of elements--in the html markup. On this page, for instance, the <h1> element at the top of the page that reads "Lesson 4" is a lovely shade of aqua and is rendered using the Open Sans font. In order for that magic to happen, I had to create a CSS rule that selected all <h1> elements on the page and then declare that I wanted to apply that particular color and that particular font style to them. I'm using an external CSS style sheet, so, my rule looks a bit like this:

h1 {
    color:#88CBCE;
    font-family:"open sans";
}

While we're here, let's take a minute to parse this out and pick up the syntax.

  • h1 - This is the selector. They are case sensitive and come in many different flavors. In this case it references a particular html element, or type, that I want to style. There are many types of selectors, which we'll get to in just a second..
  • The declaration block - The stuff inside the curly braces is the declaration block, which contains one or more delcarations separated by a semicolon.
  • The declaration itself is divided into two sections separated by a colon.
    • property - The first half of the declaration. This is the property of the html element selected that we want to change. In the example, I want to affect change on two properties of the <h1> element. Specifically, the color and font-family properties.
    • value - The second half of the declaration. The specified properties already have default values. I want to change those default values to these values.
  • curly braces - The declaration block is surrounded by curly braces. This effectively sets the scope of the CSS rule-set.

FYI: Most of the CSS in this lesson is written in the context of an external style sheet. External stylesheets are files separate to your html file and carry with them the *.css extention. The CSS file is then linked into the html file like anything else (e.g. links to different websites, email addresses, etc.). We'll cover all of this below, but--for now--keep in mind that while the context of CSS can vary, the syntax is consistent regardless of CSS type.

Combining Type Selectors

Here's where it gets kind of fun.

Type selectors target elements of a specific type. So, if I want all the paragraphs on my site to be formatted a particular way with CSS, I can do that by targetng the 'type' of element. For instance, let's say I wanted to apply a generic style to all the paragraphs on my page. It might look something like this: 

p {
  font-family:'open sans';
  margin: 0 0 10px;
  padding:.5rem;
  }

This CSS rule targets every paragraph element on the page and applies these rules to them. It can be overwritten, but with this rule i place, this is the default styling for every paragraph element to which this styling sheet is attached. I don't even have to think about it anymore as I add content.

If I want multiple elements on the page to have the same styling. I can apply the same rule-set to both by piggy-backing one type selector off another like so:

h1, h4 {
    color:#88CBCE;
    font-family:"open sans";
}

I have combined my selectors and now those same rules apply not just to every <h1> element, but every <h4> element as well.

Class selectors

If I have completely different types of elements that I want to style the same I can use the class attribute of the element to name them, or assign a 'class' to them and then apply a rule-set to that named class. As example, For this site, I have a CSS rule that effectively turns text green and bolds it. I've been using it to provide a little visual interest and emphasis when discussing the different elements (e.g. <h1>, <aside>, <header>, etc.), but I can apply that same rule to just about anything I like provided I can assign the same class to it. 

<h1 class="greenText"> </h1>

<p class="greenText"> </p>


 .greenText {
   color: #27AE60;
   font-weight:bold;
 }

Here's a h2 element with the mentioned effect

And here's a paragraph with the same. These elements may have nothing in common other than the fact that I want them both to be shaded green and bold when the page loads. By assigning them the same class attribute I can create a relationship and then exploit that relationship, applying a CSS rule to those elements that share that class name.

It doesn't matter what the 'type' of these elements are as I have grouped them together using the the class designator. Elements of the same type but not of the 'greenText' class are unaffected.

Id selectors

I have applied a style to all <h1> elements on the page using a type selector, but let's say there is one instance of that element that I don't want this rule-set to apply. I can single out a specific element by assigning it a proper id using the id attribute and then apply the style to an element with that specific id. This is a unique name for a unique instance of the element. We tend to use this more on the JavaScript side of things, but we can use it with CSS as well. 

<h1 id="default"> </h1>


#default {
      color: #000;
      font-family: "Helvetica";
    }

Descendent selectors

You can target element types (i.e. type selectors), groups of unrelated elements (i.e. class selectors), and specific named instances of elements (i.e. id selectors). You can combine type selectors and apply the same rule to multiple types--and, truth be told, multiple classes and ids as well. You can also target the nested elemnents of a container element. Let me explain:

Let's say I have a navigation menu to which I've applied some styling. I've made it to be a nice shade of dark blue with white text and some padding to give it some bulk. If I use the unordered list trick from previous lessons, every list item option will have that default list marker, yeah? It might look something like this:

  • Home
  • News
  • About
  • Contact

Now, I want those markers as default styling in every other unordered list I'm going to put on the page. I just don't want them there in any unordered list that's in a <nav> element. I could go and find every <nav> element on every page and manually get rid of those markers or I could apply a style using CSS that targets any unordered list inside a <nav> element.

nav ul {
   list-style:none;
}

The <nav> element is a parent, or container element. The <ul> element is a descendent element. To target the descendent element we use a descendent selector. You might also see these referred to as nested or children (although--technically--that has a different and very specfic definition).

I can also target descendent elements of specific classes or ids

.navigationMenu ul {
   list-style:none;
}

pseudo classes and elements

We're not limited to styling the elements as they are static on the page. We can also style any 'special states' of an element as well as specific parts of an element.

pseudo classes

I'm sure you've noticed this already, but have you ever hovered over a link or button and had them change stylistically? Maybe the element changes when you hover over it like a button that inverts colors or maybe there's a difference in the color of links that have been visited and those that haven't or maybe a form control has a nice little box that pops up around it when you click on it. CSS allows for not just styling the elements overall, but also allows for styling specific 'states' of an element as well using pseudo classes.

The most common ones are the ones that deal with links (e.g. :visited, :hover, :active), the ones that pull attention (e.g. :focus), and the ones that specify elements in a particular position (e.g. :first-child, :last-child, :nth-child).

If you remember a few lessons back when we were styling tables, there was one table that had every other line shaded gray. That was done using a CSS pseudo class selector, specifically this one:

#table1 tr:nth-child(even) {
        background: #ebebeb;
      }

This rule every even table row element in a section and shades it a nice, subtle gray.

There is a whole list pseudo class selectors to use. I've included a list on the references for your edification.

Pseudo elements

Pseudo elements are a bit different in that they don't select any 'real' element on the page. :before and :after are the two most common and are used to insert specialized content either before or after a particular element. Context Matters

Where you apply these rules is just as important as how you apply these rules. Context matters. The 'cascade' of 'cascading style sheets' means inheritance and precedence, so, let's take a real quick walk through that park before we talk about the different types of CSS

Inheritance

This one is easy, so, we're not going to talk about it too much. Just like you have inherited certain characteristics and traits from your parents and your grandparents and your grandparents parents, and so on up the line, descendent elements inherit styles from their parent elements. If I apply a font-family attribute to the <body> element, then every element nested, or inside that <body> element has that font-family attribute applied to them as well by default--even if you don't see it in the CSS. The style will cascade down through the descendent elements. Well, some styles will cascade and some will not. A cascading style that shades the background of every element and slaps a thick, black border around them would be...a hot mess, so, not all styles are inherited. Styles that generally apply to text (e.g. font family, font style, letter spacing, line spacing, etc.), tables (e.g. border spacing), and lists (e.g. list style types) can be inherited.

You can certainly override this styling with another font-family attribute applied directly to the nested element as styles take precedence based on proximity to the element affected.

Precedence

Precedence--for those of you unfamiliar with the term--means 'priority of importance.' In the context of an ER, the patient gushing from 87 different stab wounds takes precedence over the patient whose 'feeling a little under the weather.' In the context of the Oval Office, the President of the United States takes precedence over...well, just about anybody else in the room. In the context of a theme park, those disgusting and disreputable folk who buy the line-skip passes take precedence over those honest, salt-of-the-earth regulars waiting in line for Space Mountain.

The same is true of rule-sets in CSS. Full disclosure: I'm going to cheat a bit here for brevity's sake (and because I'm a little lazy), so, the examples shown aren't exactly the methods I'm using to get the proper end results. Just pay no attention to the man behind the curtain and you'll be fine.

h1 {
  color: orange;
  font-size: 36px;
}
h1 {
  color: red;
}

In this example, the counterfeit presentment of two brothers. Two CSS rule-sets that apply to the same type element. Which one do you think will take precedence? What's the end result?

Oh, whatever will I be?

In CSS all styles 'cascade' from the top of the style sheet to the end. The rule-sets are applied as the file is read by the browser, which means our little <h1> element has turned out red. It's important to note, however, that even though it might not look it, the first rule-set was applied and was overwritten by the first. We see evidence of this in the sizing. Our <h1> element is exactly 36 pixels high on the line. The 'font-size' declaration was applied and sticks because there was no second declaration to override it in the second rule-set.

The same precedence, or 'cascade,' applies inside individual selectors as well.

h1 {
  color:red;
  color: orange;
}

Now I'm orange!

This might be a little confusing, but precedence applies to selectors too. Every selector in CSS has a 'specificity weight.' We've talked to this point of three types of selectors: type, class, and id. Each of these selectors has a specificity weight.

  • type selectors have a specificity weight of 0-0-1
  • class selectors have a specificity weight of 0-1-0
  • id selectors have a specificity weight of 1-0-0

You can dig around in this for days, but the gist is id selectors have precedence over class selectors have precedence over type selectors.

If an element has a rule-set applied to it through its type AND through a class name attribute, the class name attribute rule-set will take precedence over the type selector rule-set and be applied exclusively, assuming there are conflicting rules. If an element has a rule-set applied to it through an id selector AND through a class or type selector, the id selector takes precedence and that rule-set is applied exclusively, again, assuming there are conflicting rules.

One last thing before we hit the different types of CSS.

While an element can only have one type or one id, it can have mulitple classes assigned to it.

<h1 class="greenText, redText">Generic Heading Text </h1>

You guessed it, precedence is a thing here too. In this case, whichever class selector is defined later in the style sheet takes precedence (like with our first example in red). It doesn't matter which class is listed last in the html markup, what matters is the order in the style sheet. If 'greenText' comes last in the style sheet, it will take precedence over 'redText' even though it's listed first in the html markup.

CSS Types

There are several types of CSS that can be applied to a page and--just like with all things CSS--precedence matters here as well. Here's a very brief rundown before we move on to the next lesson.

  • Inline style sheets - These style rules are applied directly to the element itself inline on the page. These rules only affect the specific element in which they are included. 

    <h1 style="color:blue; font-weight:bold;" > </h1>

  • Embedded style sheets - also referred to as internal styling. This type of CSS is offset in the <head> element of the page itself. These rules apply to all elements on the page in which the style sheet is included.

    <head>
    <style>
      h1 {
        color:blue;
        font-weight:bold;
      }
    </style>
  </head>

  • External style sheets - These are completely separate files that are linked in the html markup (usually in the <head> element). These rules apply to every page to which the style sheet is linked.

    In the html:

    <head>
  <link href="css/custom.css" rel="stylesheet">
</head>

    And the style sheet markup itself:

     h1 {
      color:blue;
      font-weight:bold;
    }



With the different types of style sheets there are matters of scope and precedence to consider. Each style sheet has an area of influence, or scope, right? The inline style sheets only affect the single element, while external style sheets affect every element on every page in which that external style sheet is linked. If there are conflicting rules (e.g. if a <h1> element has an inline CSS rule and a CSS rule applied via an external style sheet), the style sheet most directly relative to the element will take precedence. External style sheets have the greatest reach as they can affect multiple elements on multiple pages, but they are the lowest on the totem pole as far as precedence goes and can be overridden by the other two types. Inline style rules can only affect the element to which they are attached, but they take precedence over the other two types of style sheets because of their close relative proximity to the element affected. Best practice in modern web development is to use external style sheets exclusively and then work from external style sheets down to inline styles--if you absolutely, must-have, positively need inline styles at all (and you will, but, you know, use them sparingly).
