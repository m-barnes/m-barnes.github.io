Lists, tables, & forms

Organization is Key

So, just in case I haven't hit this hard enough already, I'm going to mention it one more time that html is about content organization. It's structure. It's framework. It's scaffolding. It's...well, you get the point, right?

Most html elements are structure-centric. They are designed to provide the foundations on which you can build and drape all your pretty, pretty content on. Some of them are more useful than others depending on the context and your need as a developer, but all of them have a purpose and that purpose is to provide structure (I'm doing it again, aren't I?).

This go round we're going to talk about three organizational constructs we can use to organize and display our stuff. They're big ideas and we've got a lot of ground to cover. So, while it's a lot ot trudge through, stick with it and make sure you understand what's going on because--trust me--they're pretty important. Like I've said, each has a purpose, each has a part to play, and with some of them we can use to do some pretty cool stuff.

Lists, tables, and forms, oh my!

Lists

Lists--taken at face value--aren't much to really talk about. Lists are a part of everyday life, yeah? We make 'to-do' lists to plan out our weekends, our GPS app prints out our directions in list form, and recipes are lists of ingredients complete with directions to cook delicious, delicious food stuffs. Lists are everywhere, so, it makes sense that they're everywhere online as well.

When we want to use a list on our website we have some options. Structurally, we're limited to a handful of different types of lists (e.g. ordered lists, unordered lists, and description lists). We can do some neat stuff when styling them, but--structurally--there's only these. So, before we move too far along, let's take a look at them.

Unordered Lists

Unordered lists are lists of related items in which the order doesn't really matter. Unordered lists are block-level elements, which means they block out all other content on the line once they are instanced. Unordered lists are bracketed by the <ul> tag (unordered list) and then each item on the list is bracketed by a <li> tag (list item).

<ul>
  <li>Item 1</li>
  <li>Item 2</li>
  <li>Item 3</li>
</ul>

And the result on the page:

  • Item 1
  • Item 2
  • Item 3

By default, the browser does a couple of things to offset our list stylistically. Most browsers--and your mileage may vary--add a vertical margin and some padding to shift the whole thing to the right a bit. They also offset the list items by adding a nice little graphical element (called a marker) before each item. Depending on what 'level' that list is on, in most cases that marker is a single, solid dot, which we can change or omit using CSS, or cascading style sheets (we'll talk about that later, so, stay tuned).

Ordered Lists

Ordered lists are--you guessed it--lists of items in which order does matter. Ordered lists are also block-level elements and are bracketed by the <ol> tag (ordered list). Just like unordered lists all items in an ordered list are bracketed by a <li> tag.

<ol>
  <li>Item 1</li>
  <li>Item 2</li>
  <li>Item 3</li>
</ol>

And the result on the page:

  1. Item 1
  2. Item 2
  3. Item 3

Again, like an unordered list, the browser does a couple of things to our ordered list to make it distinct on the page. The big difference here, however, is the inclusion of some numbering of some sort because with ordered lists order matters.

Ordered lists also have some other unique attributes to them with which we can play.

First of all, we can use the type attribute to change the type of numbering in the our list. We can have ordered lists start with traditional Arabic numbers, letters (both uppercase and lowercase), Latin numbers (I, II, III, IV, etc.), etc. Just add the type attribute inside the <ol> tag like so:

<ol type="A">

In the context of our generic list that gets us this:

  1. Item 1
  2. Item 2
  3. Item 3

We can also use the start attribute and start our list at any number we want. You want to start a list at 45? You can totally do that. Just add the start attribute inside the <ol> tag like so:

<ol start="45">

Which nets us this on the page:

  1. Item 1
  2. Item 2
  3. Item 3

We can also throw the list into reverse, you know, if you want. We just add the reversed attribute into the <ol> tag.

<ol reversed>
  <li>Item 1</li>
  <li>Item 2</li>
  <li>Item 3</li>
</ol>

Which gets us this:

  1. Item 1
  2. Item 2
  3. Item 3

We can also throw random numbers into the list just for kicks. We add the value attribute into the individual list item element. If left alone, the list will continue counting up from the inserted value.

<ol>
  <li>Item 1</li>
  <li value="983">Item 2</li>
  <li>Item 3</li>
</ol>

And--as expected:

  1. Item 1
  2. Item 2
  3. Item 3

Description Lists

A lesser used type of list is the description list. They have a slightly different format and are usually used to outline lists of terms and their descriptions like what you'd find in a glossary or index.

A description list may have several terms and descriptions, one after another. They may also have multiple terms per description and multiple descriptions per term. A single term may have multiple meanings and need multiple descriptions. By extension, a single description may work for multiple terms.

The description list is bracketed by the <dl> tag (descrption list), each 'term' is bracketed by a <dt> tag (description term), and each 'definition' is bracketed by a <dd> tag (description definition). The description term and the description definition that follows it correspond to one another, so, the <dt> element must come before the <dd> element. Order of these elements is super important, yeah? 

<dl>
    <dt>study</dt>
      <dd>The devotion of time and attention to acquiring knowledge on an academic subject, especially by means of books</dd>
    <dt>design</dt>
      <dd>A plan or drawing produced to show the look and function or workings of a building, garment, or other object before it is built or made</dd>
      <dd>Purpose, planning, or intention that exists or is thought to exist behind an action, fact, or material object</dd>
    <dt>business</dt>
    <dt>work</dt>
      <dd>A person's regular occupation, profession, or trade</dd>
</dl>

Which gets us all this:

study
The devotion of time and attention to acquiring knowledge on an academic subject, especially by means of books
design
A plan or drawing produced to show the look and function or workings of a building, garment, or other object before it is built or made
Purpose, planning, or intention that exists or is thought to exist behind an action, fact, or material object
business
work
A person's regular occupation, profession, or trade

I've offset the descriptions from the terms to better illustrate the difference between the different elements, but in practice the margins of the descriptions would align to the left by default.

Nesting Lists

So, one of the more powerful features of using lists is their ability to be nested. This makes them terribly useful for developers when dealing with site navigation, which we'll talk about in just second.

Before we do, however, let's take a look at some code

<ol>
  <li>Walk the dog</li>
  <li>Fold laundry</li>
  <li>
    Go to the grocery and buy:
    <ul>
      <li>Milk</li>
      <li>Bread</li>
      <li>Cheese</li>
    </ul>
  </li>
  <li>Mow the lawn</li>
  <li>Make dinner</li>
</ol>

As you can see, we've nested an unordered list inside an ordered list

Let's see how this plays out on the rendered page.

  1. Walk the dog
  2. Fold laundry
  3. Go to the grocery and buy:
    • Milk
    • Bread
    • Cheese
  4. Mow the lawn
  5. Make dinner

A few things to note here, okay? The first is that nesting lists is tricky as unwanted styling can pop up unintentionally if you're not careful, so, mind your opening and closing tags. The second is that as you nest lists--particularly unordered lists--the styles for the markers will change. You might have noticed that the unordered list here is delineated by open circles instead of the solid dots like our previous example. This is because the unordered list is nested within an ordered list. This unordered list is at a different 'level' in the list hierarchy.

Fortunately, we can control all these default styles with CSS, which is where it becomes particuarly neat for us as developers. I've provided a list of some sample CSS you can throw into or use in conjunction with your code that gives you some styling options. You can find it with our old friend, the reference page. You'll understand it a bit better once we start digging into CSS, but since we're here, I thought I'd go ahead and include it for you.

Tables

Tables used to be the thing when building websites back in the day. There has always been a need to be able to organize tabular data--stuff that would look good in a table--on the internet, but they're also pretty good at managing and organizing data of all kinds. Unruly photo collages, misbehaving text, wandering menus, etc. were all powerless in the face of the all-powerful table and, so, tables started popping up in code...everywhere.

In the beginning, this was all right because developers were dealing with some simple and pretty fixed variables as far as design and display were concerned. Even as websites started to get more and more complex and user demand drove the need for more sophisticated design and functionalty, tables were still pretty useful. All kinds of design styles and attributes (e.g. 'colspan,' 'rowspan,' size percentages, nested tables, etc.) were being used heavily in development.

The problem is--and I like using analogies, so, hear me out--it became a 'jenga' problem. Tables were never really meant to do these kinds of things, right? The benefits of using them this way were a by-product of the inherent properties of the element. You start stacking this incredibly dense and heavy stuff on top of a weak foundation and the center cannot hold. Inital setup wasn't too bad. It was complicated, to say the least, but starting from scratch was usually better than trying to modify code in-situ--espcecially if you didn't put it together originally, right? Digging through lines and lines of code looking for the one thing--stuck in the middle of the middle of a bunch of tables--that you wanted to change was time-consuming. It was also frustrating because even if you managed to find it, a change often set off a chain-reaction of formatting issues that you then had to fix.

I'd like to take just minute to issue a word of warning to the wise. You know, in case you've missed it. Tables are powerful, easy to use, and you will be tempted to use them to wrangle unruly content--especially after fighting for hours on end with your code to get that one stupid picture to fall in line and sit off to the right without covering up that text. I mean, holy crow, what does it take to do that? I've done everything I can think of and it just sits there on the page...mocking me. I hate it.

That is to say, you will be seduced by the power of tables and might think to use them inappropriately in your work because of frustration or apathy, or whatever, but do not give into anger and hate for that way leads to the Dark Side. Tables are powerful, but they come at great cost. It can be more difficult to do what you want to do without the benefit of using tables inappropriately, but while learning to do these things the right way might be harder, you will reap greater benefit in the long run and grow into a powerful Jedi in your own right.

So, before we get into tables proper, let's set some ground rules on using tables and get into some reasons why you shouldn't use them inappropriately for content layout:

  • Tables shouldn’t be used for general layout purposes.
  • Tables are not accessible, meaning screen readers have a real problem with them.
  • Tables are tricky to use, especially in complex designs.
  • Tables shouldn’t be used for general layout purposes.
  • Tables are inflexible. Generally, tables are slower to load and depending on how they are formatted, can dramatically change the layout of your content on screens of different sizes.
  • Tables shouldn’t be used for general layout purposes.
  • They don’t always print properly. This isn’t always an issue for the casual user, but in the corporate climate, for instance, the ability to print content is still kind of a thing, which means you have to ensure that what you see on the web page is what you get on the printed page.
  • And, finally, tables shouldn’t be used for general layout purposes.

Truth be told, using tables for layout has been kind of a no-no since HTML 4.01--although that still didn’t stop everyone from doing it--and while HTML5 has changed the rules a bit, the specification still states:

Tables should not be used as layout aids.

So, there you go. Tables are still perfectly acceptable for displaying tabular data, by the way. Feel free to use them for data that ‘naturally,’ belongs in grids (e.g. charts, some lists, inventories, etc.), but--and I feel this is important to mention--you shouldn’t use them for layout purposes. If you need to wrangle non-tabular content, There are other means. Sometimes using those means is a bit more difficult, but walk the path, Padwan, and you'll be better off further on down the road.

Tables are for data that naturally fits into columns and rows. Html5 has a number of elements available for defining and providing structure to the data. Like our lists example, we're going to take look at some basic html code for tables, build on it a bit, and then make it shine using some CSS.

Forms

Last, but not least, we have forms.

Forms are an essential and ubiquitous part of the internet these days. As developers we use them for all kinds of things from capturing data from users (sign up here!), to processing requests (I'm feeling lucky.), to controlling application function and access (Username and password, please.). If there is interactive functionality built into a website, however, small that functionality might be, chances are it's been framed out using form elements.

As frontend developers we need to know how to build forms, so, we're going to discuss how to use html to create some mark up, the roles and uses of some common form elements, and a bit on styling those forms using CSS. Later on we'll even wax poetic on using JavaScript to validate the data input into those forms. What we won't talk about is how to process those forms, yeah? Processing that data, or determining what we do with that data, once it's captured is the business of a backend developer. They use stuff like PHP, python, SQL, Ruby and whole host of other made-up sounding languages to deal with those things on the server side. For our purposes all the information we collect is going to be packaged and sent to a generic email address.

Forms are added to the page using a <form> element. This element identifies where on the page control elements, or form fields, are going to appear. They define the borders of the form itself and help keep all the form elements in their designated areas while at the same time keeping all the wandering riff-raff out.

Like any other element we can apply attributes to them. The two most common are the action and the method attributes.

action - This attribute contains the url to which the data collected in the form is going to be sent for processing. If I'm Googling something, for instance, this is part of the form they use to process my search.

Now, this is Google, so, it's pretty extensive, but--in a nutshell--they are going to take the data from the search box and then append it on to the end of a url that will be sent to a Google server for processing. The 'action' in our screenshot is '/search' which is the tail-end of the proper Google search string--https://www.google.com/search The form data will be formatted and tacked on to the end of it to be submitted. Whatever is found will be returned in the browser window.

method - This attribute references the http method that should be used by the form when submitting data for processing. There are two basic 'methods' used:

GET - A 'GET' method request is pretty public. The data--in this case it's data from a search box or query--is simply tacked on to the url provided by the action attribute. It's cached by the browser for later use, it can be bookmarked by the browser, saved in a server log, and is sent across the internet in plain text out there in the open for everyone to see. The 'method' in our screenshot is a GET request. This means that whatever you put in that search box is going to be appended to the url https://www.google.com/search. When the search results are returned that modified url is going to be in the address bar at the top of the browser. The search keywords will be encoded into the returned url. The browser will cache it for later use and somewhere--I'm betting--it's saved to a traffic log at Google HQ probably tied to a whole bunch of other identifying information like an IP address, time/date, etc.

As example, here's a recent Google search of mine.

https://www.google.com/#q=how+to+get+rid+of+blood+stains+fast

Guess what I was searching for? You guessed it--I was searching for an easy no-bake cookie recipe.

GET requests are primitive. The only real way for data to pass via a GET request is through the url, which provides a very, very limited amount of space in which to pass data. Consequently, GET requests should really only be used for retrieval of simple, non-sensitive information like when querying a search engine or requesting a url or whatever. Developers like to say that GET requests should be idempotent (because it makes us look wicked smart), meaning you could do it a million times and the server will do the same basic action and return the same basic results each and every time. Google uses a GET request because your search keywords aren't considered sensitive information and even though the results themselves may differ, the action to return those results is exactly the same. A rose is a rose is a rose regardless of where it blooms.

Using the 'GET' method means I--as a developer--can do more stuff with that data, but it comes at the expense of privacy, so, it might not be a good idea to use it on a form asking for sensitive personal information like credit card information or a medical history. Not that you could, but you get the point, right?

POST - A 'POST' method request is a bit different. It works the same--for the most part--but the data isn't sent via the url, but in body of the request (a GET request has no body). This means there is exponentially more space available to send data back and forth between client and server (e.g. account creation forms, records requests from databases, etc.). An additional benefit is that--because the data is sent via the body of the request--the data can be encoded, encrypted, or otherwise obfuscated to ensure that should that information get into the hands of the bad guys, those bad guys can't really do a whole lot with it. The data sent via a POST request isn't appended on to the url, it isn't cached by the browser (unless you specifically tell your browser to do it in the name of convenience), and your data isn't saved to some server log somewhere at the time of transaction. If you've ever filled out a form, hit the back button, and gotten that little box asking you to confirm resubmission, you're dealing with a form that uses a POST request. Unless you specifically authorize it to do so, your browser will only save the data from your submission until the browser is closed out and only on your side. The record of your transaction may exist (Yes, I can confirm that Bob was here and told me some stuff), but the contents of that transaction will not exist beyond the time it takes to complete that transaction (No, I have no idea what we talked about).

Creating Forms

Now that I've bored you to tears with all this, let's talk about actually creating some forms using some of this stuff.

We've got our <form> opening and closing tag, we know where we want to send the data we're going to collect, we know what method we're going to use to send it, so, let's actually collect some stinking data.

The single most important form element is the <input> element, which uses the type attribute to define what type of information is captured in the control. Using the type attribute we can shape our inputs to accept a single line of text, multiple lines of text, create radio buttons, checkboxes, drop down lists, create submit buttons, etc. <input> elements are self-contained, or self-closing, which means there is no closing tag. The value of the element is provided for by its attributes and the values that correspond with those attributes. It's also best practice to provide each <input> element with a name attribute. This ties the data and the form field, or form control, together. Having this connection will make things easier for our friends on the backend when they go about their business of working with the data that the form has collected and submitted.

So, without further adieu, let's take a look at some of these <input> elements. Examples follow the established format: code first → rendered results.

Drop-down lists - Drop-down lists are an excellent way to condense a long list of options into one, relatively small control. Radio buttons have had their day--and will continue to do so, I'm afraid--but drop-down lists are, in most cases, the better choice, especially on sites designed with mobile in mind (and they all really should be designed with mobile in mind). Drop-down lists are small, compact, and are easy for most users to understand.

To create a drop-down list you'll need two new elements: the <select> and the <option> elements. The <select> element wraps all the menu options and the <option> element marks up each menu selection.

The name attribute is tied to the <select> element and the value attribute is tied to the <option> element.

Each <option> element wraps the text of each individual selection in the menu that is shown to the user. Just like with radio buttons and check boxes, you can pre-select an option in a drop-down list using the similar-meaning boolean selected attribute.

Whose your favorite Avenger?

<select name="avenger">
  <option value="Captain America" selected>Captain America</option>
  <option value="Black Widow">Black Widow</option>
  <option value="Hawkeye">Hawkeye</option>
  <option value="Hulk">Hulk</option>
  <option value="Iron Man">Iron Man</option>
  <option value="Thor">Thor</option>
</select>

Whose your favorite Avenger?

The only valid answer is 'Thor,' by the way.

And--in a similar vein--you can have multiple selection with drop-down lists. Just add the multiple attribute to the <select> element.

Whose are your favorite Avengers?

Form buttons - You need a way to 'fire' the form submission event and send the data collected on its way. Submission buttons are typically used for that. The value attribute is used to define the text that shows up 'on' the button. 

<input type="submit" name="submit" value="Submit">

You might also have a 'reset' button to clear out all the data on the page so that the user can start over. You don't see them too much these days, but I've included it anyway because I feel like the lesson isn't long enough already.

<input type="reset" value="Reset">

Organization Redux - Electric Boogaloo

Almost, but before you go (assuming you haven't left in disgust already), let's talk about form organization for just a bit.

Form fields are great, but most of them don't come with any inherent labeling, which is a problem. I mean, how does the user know what to do if they have nothing there to guide them? In our examples, I had a few labels here and there, but they weren't proper labels because I didn't want to confuse the issue. Now that you've made it thus far, let's learn how to do it properly.

Labels provide captions or headings for form controls. By tying them together we help streamline data processing and make our forms 'accessible,' which means a better experience for users including those that use assistive technologies to view our content. Here's how we do it.

<label for="username">Username</label>
<input type="text" name="username" id="username">

The for attribute of the <label> element carries the same value as the id attribute of the <input> element effectively connecting the two elements together. If you click the label, the input field automatically grabs focus. Go ahead, give it a try.

So, labels are good. Now the user knows what kind of action we expect from them and what kind of data to input. However, in particularly complicated forms (or maybe if you just have a particular aesthetic bent) it might be beneficial to take our organizational compulsion a bit further.

Fieldsets group labels and controls into little organized clusters of goodness. In a way similar to the <section> element, the <fieldset> element is a block-level element--complete with border that can be modified--that divides our form up into nice, pretty, easily digestible chunks for the user.

<fieldset>
  <label>
    Username
    <input type="text" name="username">
  </label>
  <label>
    Password
    <input type="text" name="password">
  </label>
</fieldset>

See how nice that looks? Let's make it look a bit nicer, yeah?

Legends provide a caption, or a heading for the <fieldset> element. The <legend> element wraps the text for display describing the fieldset and should be included immediately after the <fieldset> element.

<fieldset>
  <legend>Login</legend>
  <label>
    Username
    <input type="text" name="username">
  </label>
  <label>
    Password
    <input type="text" name="password">
  </label>
</fieldset>

Login

Placeholder is an html5 attribute that provides a hint or tip within the input control to subtly give the user an example of the kind of information they should be providing. It's not advisable to use them in place of a label, even though you will be tempted because not all environments handled the placeholder attribute in a consistent manner. Unlike the value attribute, the placeholder attribute disappears once the input is selected and the user starts typing.

<label for="email">Email</label>
<input type="email" id="email" name="email-address" placeholder="firstname_lastname##@asumh.edu">
