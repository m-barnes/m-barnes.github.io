Lesson 2

Structure

Take a minute to think about all the stuff you read in all the different formats in all the different types of media in...well, you get the point, yeah?

Think about what makes them readable. Newspapers, term papers, books, lists of 'things to do,' class schedules, this website, etc. are all readable--to varying degrees--because of the structure in which that content is laid out.

Titles, headings, paragraphs, offset images, etc. are all used to define the structure of content, which--by extension--promotes readability. Readability eases reading effort and speed for the reader, which means better comprehension, which means better retention, so, structure is pretty important, yeah? As a web designer or developer (and there are differences) you want to not only make sure that people are looking at your product, but that they're sticking around to hear--and understand--what you're trying to say.

I may have mentioned it a time or two in the last lesson, but html is all about structure. It helps us control content to better convey our message to the masses.

Let's take a closer look at some actual code and how this all works.

All clearly defined by spacing and structure. Your eye naturally moves from one bit to the next automatically and you know the intent and purpose of each grouping without having to really think about it.

Let's break this down section by section, tag by tag:

From the top

The first section is what I like to think of as 'staging.' The content here, the elements we use, aren't for the public, at large. This area is about setting the stage, getting the space ready for the public. We start with a DOCTYPE declaration. This is an empty tag that sits by itself and separate at the very beginning of the document. It's purpose is simply to let the browser know that the file it's fixing to work through is, indeed, a html file and that the browser should treat it as such. In days past, you'd have also included a document type definition, (DTD), to let the browser know specifically what particular framework was used to structure the document and might have included a referencing entity (the example below does not). It might have looked something like this:

<!DOCTYPE html PUBLIC"-//W3C//DTD Xhtml 1.0 Transitional//EN">

In html5 there is no DTD, so, the DOCTYPE declaration is short and sweet. In the example below, 'html' is lowercase. The formal syntax is uppercase, but for compatibility purposes (just to make sure) I like to keep it lowercase as both cases are treated as valid in html5.

<html lang="en">

The next line is our opening html tag. Technically, it's not required in html5, but, again, for purposes of compatibility, it's generally a good idea to include it--at least for right now. Everything we do on the page from here on out will be inside the opening and closing tags of this element. Every bit of background work, every bit of content, every navigation menu, paragraph, heading, etc. will be between the <html> opening tag and the <html> closing tag. The lang="en" attribute is markup solely included for search engines and browsers. It's not required, but is good practice if the element is included, so, I've included it here.

The <head> element is next and used as a container for all 'head' elements (title, meta information, scripts, referencing styles, etc.). In html5, the element can be omitted as it's not required and a lot of developers leave it out, but, again, I consider it good practice as a matter of personal preference, so, I include it. This one feel free to leave out of your own work, however, if you like.

  • <meta>- The metadata section (data about data) includes information relevant to search engines (keywords), to the browser (character sets, author) and to other web services. In our example, we've actually got two <meta> elements. The first has an attribute defining the character set ("utf-8") and the second has information on how to display the page (name and content). The content attribute we will talk about at length later when we discuss display properties and responsive design. Generally speaking, the attribute information in this element deals with content scaling and controlling page dimensions for display.
  • <title>- Defines the title of the document and is REQUIRED in all html documents. Its purpose is to define a title for the page, which will be used for the browser tab, or when it's added to a browser's favorites, and for search engine results.
  • <link>- links to any secondary resource, typically, an external CSS stylesheet, which is exactly what our example features. In this instance, we're refencing an external stylesheet located in the 'css' folder with the name of 'custom.css.' This is a relative link and we'll be looking at them later on in this lesson when we talk about 'paths' so, stay tuned.
  • <style>- if we didn't link to an external stylesheet, but still wanted to style elements on the page, this is where you'd include those rules. Style rules in the 'head' of the page are referenced as an embedded stylesheet. When we move the discussion to CSS, there is a substantial section on the different types of stylesheets and their uses. I've just included it here for illustrative purposes.
  • <script>- if you had some JavaScript you wanted to run on the client side (again, we'll get to it, I promise) this is where you would include it.

With the closing </head> tag we define the end of the 'staging' section. Search engines have their bit to index, the browser has the parameters it needs to start generating some output, and the end user doesn't see any of it. We have effectively set the table for the content buffet to follow.

Meat and Potatoes

The bulk of what the end user sees is located within the opening and closing <body> tags. The element delineates main content that's usually specific and particular to that page, so, content specific to a page that features an ordering form, or to a page that features an 'about us' section, etc. is typically found in the <body> element. Our example has a few different types of elements that are used in the main content section of a page. Let's go through the list, yeah?

  • <nav>- The first of our html5 specific semantic tags designed to offset navigation features on the page. Our example is a single page, but in instances where the navigation menu is meant to carry over consistently across multiple pages, a developer might consider surrounding the <nav> element with the <header> element meant to indicate header type information. This should not be confused with the <head> element.
  • <ul> and <li>- These elements' stock and trade are lists. 'ul' indicates an unordered list, which--by default--are offset with bullet points. In our example, the unordered list is used to create a navigation menu, so, the bullet points have been 'removed' using CSS. I've also included a style attribute that makes the list items display horizontally. This is an inline style and, again, is only included for illustrative purposes. The 'li' indicates a list item. Each item in the list gets its own set of opening and closing tags to separate them from one another.
  • <div>- I mentioned this one before, but the 'div' tag simply indicates a division or section of content. I could have just as easily--and probably should have--used the section element. The 'div' element is really our jack-of-all-trades element. We have a ton of new semantic elements in html5 that we can use, but there are still some gaps, yes? The 'div' element is meant to fill those gaps and provide a bit of flexibility and adaptability to our html code. This particular example features an id attribute, which specfically identifies this particular 'div' element, which we can use to style it or provide a bit of interactivity using JavaScript.
  • <hgroup>- Heading group. A group of heading elements. Some people use it; some people don't. Your mileage may vary.
  • <h1> - <h6>- Heading elements of varying size. 'h1' is the largest; 'h6' is the smallest. These are block level elements meant to indicate important content or important shifts in thematic content.
  • <aside>- 'Aside' is meant to indicate content that is related to main content, but needs to be offset in the display. Sometimes it's contact information, sometimes it's a targeted ad, sometimes it's secondary navigation (like the jump menu to the right). This particular 'aside' has a style attribute. This also features inline styling (something we'll talk about when we get into CSS) and features a couple of rules specific for its display.
  • <p>- The last--and probably most used--element is the paragraph element. It's pretty self-explanatory, so, I'll just leave it at that.

The bottom of the barrel

The footer is area reserved for information or features you want to carry across multiple pages of your website. It often works in conjunction with the <header> element to frame the changing content of each individual page. Any sort of content can go in the footer, but it's typically stuff like contact information, logos, copyright information, links to support services, etc. In our example we have a couple of new elements to mention:

  • <h4>- Heading level/size 4. Kind of mid-range importance.
  • <address>- Address information. Here it throws the whole thing in italics, but we could just as easily feature a logo or some other company or personal brand identifying element.
  • <br>- Line break. It's like hitting return on a text document. It immediately takes the next bit of content--whatever that content is--and moves it to the next line.
  • <hr>- It's not featured in our example, but you might also offset the footer using an <hr> element, which would put a horizonal line across the page. These are used to break up or separate thematic content.

It's important for you to understand that what's included in the example is not the totality of elements that are available to you for your use. I've included a reference page that has some of these different elements in their different contextual catagories, but even this is not a full and comprehensive list. When you start developing pages of your own, there is a very good chance that you'll have to do some outside research to find the right element for your use and context.

This is good.

Language--and all languages not just those that help us put together websites or other software--is an ever evolving thing and not everyone agrees on the syntax or its use contextually. Diversity of opinion and experience matters and while it might seem ridiculous to have to wade through some of this stuff, it will force you to examine these things critically, which will make you a better developer in the long run as you form your own opinions and preferences.

Structural Concepts

We're almost there.

Semantics and Styling

In the early days of web development pages were constructed using 'divisions.' <div> tags were everywhere and while most developers were pretty good about providing comments and including identifying attributes with the element to help navigate the code, it was still super easy to get lost in the markup, especially on larger, more complex projects. Divisions--by themselves--provide no semantic value, so, determining intent--even with the additional comments and attribute information--wasn't the easiest most times.

html5 has introduced a whole host of new structurally based elements. The great thing about these new additions is that they are semantic, which means they describe the content meant to be contained inside. They can be used and reused as the developer sees fit, they are all block-level, and they all have no default or implied position or style. This means we can put them all over the place, we don't have to worry about how they'll interact with other elements, and we don't have to worry about offsetting a default position on the page or style (like with our ul list items that--by default--come with bullet points, yeah?).

We've already talked about a few of these and there are a lot more in the standards, but let's take a look at a few of them in the context of semantics and styling. The important idea to keep in mind when deciding which element--especially semantic elements--to use is context (e.g. what type of content is being described in the element).

Which element is the 'best' element? Look at the content inside and then make the decision.

<head> vs. <header> vs. <h1> - <h6>

It's easy to confuse these, but they all have specific purpose and meaning:

  • <head>- is not displayed on the page. It's used to outline metadata, titles, links to outside references, etc. It falls directly within the <html> element.
  • <header>- is a semantic element that outlines the 'heading' segment of a page. It falls within the <body> element. You could, theoretically, have multiple <header> elements on the page.
  • <h1> - <h6>- are used to indicate levels of 'headline' text on the page. 'h1' signifies the most important; 'h6' signifies the least important. They are styling elements meant to provide thematic structure to groups of text like that found in <p> elements.
<article> vs. <section> vs. <div>

Both <article> and <section> provide structure and help outline the document (e.g. this is an article; this is a section). They are semantic. If the content inside provides value to the outline of the document, you use one of these.

  • The <article> element is meant for content that can be 'independently redistributed.' If it can be separated from the page and still stand alone thematically, you use the <article> element.
  • If the content adds to the value of and is dependent on other content on the page, if it can't be separated from the page and stand alone thematically, you use the <section> element.

If the element doesn't provide value to the outline of the document or if the content is being grouped solely for styling purposes, you use the <div> element. Again, the <div> element is used as a catch-all for the content that just doesn't make sense anywhere else or if you want your style rules to have an effect on a particular set of elements (e.g. you can have <article> elements and <section> elements inside a <div> element that might be styled the same).

Paths

By this point, you might have noticed that we've been linking stuff left and right. You've seen examples of links to external CSS stylesheets, you've seen links to other webpages, you've seen links that make the viewport jump down the page to a particular section, and a whole bunch of other things. Now down below, we're definitely going to run through some exmaples of the different kinds of linking that you'll do as a developer, but before we do, we need to have a discussion on 'pathing.'

The two most common types of links are links to other pages of the same website and links to other outside websites. These links--you may have noticed--are identifed by their href attribute, which, in this case, is also known as a 'path.'

Relative Paths

Links pointing to other pages or other parts of content of the same website use relative paths. Relative paths link content based on current directory.

Think of it like this: If I want to get to Walmart and ask you for directions, what's the easiest way for you to explain to me how to get there?

You first have to figure out where you're currently at, right? Once you determine that, you start picking out landmarks and putting together relative directions (left, right, down the road, etc.). Your directions to Walmart might include something like "Well, you keep going down this road, past the library on the hill, take a left at the light, and keep going til you see Home Depot. Walmart is on the left, you can't miss it."

This is relative pathing. You have to know where you're at in the current space to figure out where you have to go. You use landmarks and relative directions to navigate from A to B. It's quick, simple, and easy to understand, yeah?

It works like that with links in html files as well. The file folder with the html file you're currently working on can be thought of as the 'home' directory. This is your starting point. Things that you link to inside the html file that is located inside that 'home' directory have positions (or directions) relative to the 'home' directory. Linking to stuff inside the 'home,' or current, directory is easy, you just include the name of the item you want to link.

<link href="custom.css" rel="stylesheet">

In a generic html file, you might find something like this. It's a link to an external CSS stylesheet called 'custom.' We're telling the browser that we definitely want to use this stylesheet and we're giving it the location that it needs to find it. In this case, the CSS file is in the same directory as the html file we're editing (they're in the same room), so, we only need to include the name of the actual CSS file itself.



If that file were somewhere else, but still within the internal framework or inside the scope of the project (inside a folder just for CSS files, for instance), then we'd have to modify that link. It would still be a relational link because we're still giving it directions in relation to the html file that's linking, or calling, it, but it would look different.

<link href="css/custom.css" rel="stylesheet">

In this case, the 'custom.css' file is in a folder separate to that of the 'home' directory or the folder in which our html file is located. When the browser loads the html file, it needs to know not just that the 'custom.css' file is being linked, or called, but also where that file is located in relation to the html file it's loading. So, we give it directions to Walmart. The 'custom.css' file is in the 'css' file folder over there.





In the beginning, it will be easier to keep all your 'assets' in the same directory (css files, images, JavaScript, etc.). Depending on your process, you probably won't have too many separate files of which to keep track.

As you move further along and your websites start getting bigger and bigger, you'll find that you're going to need to impose a little organization to better keep track of things. You might have folders full of different css files, or JavaScript files, or images, or whatever. As long as those 87 different images are considered part of 'your' website, then relational linking is the way to go.

Now, you should understand that the ease of relational linking comes at a cost (doesn't everything?). If--at any point--you move something in the hierarchy of your site (e.g. you move a css file, you move a folder full of images, you change the name of a file folder, etc.) your relational links won't work. If, for instance, you change the name of your css file folder to 'assets', then your relational link--as it's written above with the path 'css/custom.css'--isn't going to work anymore because the 'css' doesn't exist anymore. The landmark is missing.



<link href="assets/custom.css" rel="stylesheet">

To 'fix' the link, you'll have to change 'css' to 'assets' in the path. The path has been fixed, the landmark has been found, and all is well again.

Absolute Paths

Absolute paths are for links to things outside the scope of your website. Links to sites that you don't control or manage, for instance, use absolute links complete with protocol prefix and domain suffix.

<a href="http://www.google.com">Google</a>

This is a link to Google.com. It is an absolute link, including the 'http' protocol prefix and the '.com' domain suffix. Google.com is a published website (made available to the public at large) that we don't have insider access to (assuming you don't work for Google, which if you did, what are you doing here?). Anything that we want to link to that we don't have administrative access to must be published and we must use an absolute link.

To go back to our Walmart analogy: this is like Walmart's physical street address.

65 N Walmart Dr, Mountain Home, AR 72653

You can totally use this address to provide directions. It's not terribly practical, but it works. I can plug this into my phone or GPS and it leads me to Walmart in Mountain Home every single time. If, for instance, the landmarks around Walmart change (Home Depot goes out of business and is replaced by a ginormous Steak and Shake) or if they build a new road to Walmart that gets you there 3x as fast as the bypass, I can still find Walmart because Walmart's address remains unchanged.

If I link to Google.com, it doesn't matter where the actual files that make up the site that Google publishes are located, or how they're structured on the back end. Google's website is published as 'Google.com' If Google decides to up and move their operations to the Moon, as long as they publish the site using the URL 'Google.com' I can link to it and it will work every single time because the address 'Google.com' remains unchanged.

So, if absolute links can be used in place of relative links and they never change, then why don't we just use absolute path links for everything?

One of the big advantages of using relative path links is for testing. I'm writing this particular part of the site on my laptop while sitting in my recliner in my living room. I can write it, test it in my browser, fix the gazillion typos, test it again, etc. with very little hassle. My CSS is working great. It turns some stuff green, it turns some stuff blue, keeps everything looking the way I want it to look, and is linked using a relative path. If I zip up this folder and move it from my work laptop to my personal laptop, throw it to the cloud, or even upload it to a work server, it will look the same when I test it provided I don't change the folder hierarchy or rename a file.

Here's where it gets a bit...involved, but stick with me. As previously discussed, absolute links kind of require a complete URL to work, which means protocol prefix and domain suffix (e.g. http://www.google.com, https://upload.wikimedia.org/wikipedia/commons/f/ff/Carl_Bloch_-_In_a_Roman_Osteria_-_Google_Art_Project.jpg). To get or use those things you need to 'publish' the website first (e.g. upload it to a hosting service, attach a domain name to it, etc.). Even if you could see the future and build all your absolute links with perfect knowledge of what your domain name and the like would be, until it's published, that site doesn't exist as far as the internet is concerned. Those 'perfect' and absolute links wouldn't point anywhere, which means no local testing in a local development environment.

So, to test, you'd have to use a 'live' environment, which you certainly can, but it brings with a whole host of problems with it as well.

If I used absolute paths for my CSS files or image files, I would have to 'publish' the site before I could see the results of any changes I made to the code. If I made a slight change to the CSS (e.g. I want certain elements to be red instead of green), I would have to make the change in the CSS file locally, upload and publish that file to a hosting service, and go to that published site using the protocol prefix and domain suffix before I could see the changes I made in the browser. A few changes here and there and it isn't too bad, but if you're making changes left and right, that time and work starts to add up. Also, in reality, anyone with the published link would see the changes 'live' as well.

So, if you're the only one involved and you're okay with it, then run with it. If, however, you aren't the only hand in the pot and you're not the captain of the ship, then the general consensus is that relative links are used for internal content (part of your website) and absolute links are used for external content (third-party stuff you don't control).

Linking

On the page you can link to just about anything. As we've discussed, the ability to make connections between discrete and separate parts of your developed website or to resources available out there is one of the more powerful applications of the internet at large. Hyperlinks (what a dated term, yeah?) are created using an <anchor> inline level element. To make the connection between that element and the resource you want to link you have to use the href, or hyperlink reference, attribute. Whether that's an image, CSS file, JavaScript file, internal or external website, a block of text further down on the existing page, etc. that href attribute is representative of our destination.

So, let's run through a few of the ways to create links for some of the more common items you might find on a webpage.

<a href="http://www.google.com">Google</a>

This is a basic URL link. We've seen this before, but let's take a minute to break it down and parse it out.

  • <a> - The letter 'a' is shorthand for the anchor element
  • href - This is our attribute. It is the destination for our link, which in this case is a url for 'google.' This is an absolute link as Google is a third-party site as far as we're concerned.
  • 'Google' - We have the element and we have the destination, but we need some vanity text on the page to which those things can be tied. The URL itself is ugly and we don't want it littering the page unless we have to have it on the page. To counter, we create a place that we can customize that the browser will format using a universal visual cue for the end user. In this case, our vanity text reads as 'Google' This is what will show up on the page and--by default--the browswer will format it a nice shade of blue and it will be underlined as a visual cue to the end user that this text--which reads 'Google'--is a clickable URL.
  • /<a> - The closing element tag.

This is--and the examples that will follow--are basic links, but understand that anchor elements--like all elements--can have multiple attributes, many of them specific to the types of links you're trying to create.

To open the URL in a new window or tab, for instance, you'd add an additional attribute 'target="_blank"'

<a href="mailto:mbarnes@asumh.edu">Email Me</a>

This link--when clicked--opens the default email application on the system and pre-poplates the 'to' field. You might have noticed there are some similarities with our previous example. The big difference is that href attribute. In this example it's just a simple destination email address being pre-populated, but you could also pre-populate the 'subject' or even 'body' text fields of the composed email.

href="mailto:mbarnes@asumh.edu?subject=That%20thing%20I&20sent%20you&body=Did&20you&20get&20that&20thing&20I&20sent&20you?."

The separate 'subject' and 'body' are addtional parameters to the attribute. We bind them together with '?' To reflect spaces in the text requires a special 'encoding character' -- '%20' A list of some common encoding characters can be found on the references page.

<section id="top">

    ...A whole bunch of text and stuff that requires an endless amount of scrolling down the page... 


    <a href="#top">Back to top</a>

</section>

This is a link to another part of the same page. I'm sure you've seen sites with links at the bottom of a page that returns the user (or moves the viewport) back to the top of the page. This site--for instance--could have used them, but what can you do?

This link works the same, but requires a few additional steps.

The href attribute is still there and in a familiar format, but this one references the 'id' of a particular element on the page. We'll talk about 'ids' and 'classes' at length when we get into CSS next time, but--for now--just understand that an 'id' is on of a few different ways we can label and identify specific elements on the page. In this case we have a <section> element with the specific 'id,' or name of 'top' The href attribute references that 'id,' which moves the page back to the place in the code, or markup, where that 'id' is first referenced. Since that reference is at the top of a whole bunch of text, the user's view is moved back up the page to the beginning.

One last one before I let you go and pay attention because this one is a bit complicated

To this point, we've only looked at the href attribute. The hypertext reference attribute is a link that defines a relationship between the html document where the link is located and the linked resouce. It doesn't actually load the resource into the page itself. This means that when the browser loads, or 'parses' the page, the browser doesn't pause rendering the page to validate the resource. For instance, when we provide a link to Google.com the browser doesn't stop rendering the page when it gets to that link to make sure that Google.com is there or not. The browser assumes it works because we've established a relationship using the href attribute. If that link doesn't work, the browser (or rather the user) doesn't know it until that link is clicked and--truth be told--even then the browser isn't aware it isn't working because it still tries to send you to wherever that href is pointing.

The other frequently used 'linking' attribute, src, or 'source', works a bit differently. It doesn't establish a relationship, but instead embeds the resource in the document at the location of the link. 

<img src="images/Zebra.png"/>

This is a link to an image with the name of 'Zebra.png' located in the 'images' folder. When the browser gets to this particular line of markup, the render is stopped until the resource is found and loaded into the document. This is a subtle, but, oh, so, significant difference, specifically in the context of loading times.

You may have noticed that not all pages load the same, right? Some are real quick, some take a little longer, and some take longer still. The more 'embeds' that page has in the markup (e.g. images, videos, audio files, JavaScript, etc.) the longer it takes to load because as the browser gets to those 'embeds' in the markup everything stops until that 'embed' is found and loaded in.

There are some methods to offset this (e.g. the loading of JavaScript can be 'deferred' until the page parses), but this is how it generally works.
